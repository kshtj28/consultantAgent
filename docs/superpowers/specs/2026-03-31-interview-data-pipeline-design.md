# Interview Data Pipeline вҖ” Design Spec

**Date:** 2026-03-31
**Status:** Approved

## Overview

Replace the manual report generation flow with an automated Mastra workflow that triggers on every interview pause or completion. The workflow generates all report types, computes all page metrics, and broadcasts real-time updates via SSE to every page in the application.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Report scope | One gap analysis report per broad area | Matches the hierarchical broad area structure; reports improve incrementally as sub-areas get covered |
| Trigger | On every interview pause/complete, for broad areas with вүҘ1 covered sub-area | Progressive reports that refine with each interaction |
| Queuing | If already generating, set `pendingRegeneration` flag; re-trigger after current run | No data loss from rapid pause/resume; no duplicate concurrent workflows |
| Async mechanism | Mastra workflow (fire-and-forget from route handler) | Step orchestration, structured error handling, extensible |
| Architecture | Parallel map/reduce with concurrent lanes | Proven pattern (existing reportService uses map/reduce); maximizes throughput |
| UI status | Top banner + inline row badges with SSE-driven updates | Banner for at-a-glance awareness; row badges for detail |
| Manual generation | Removed | All reports auto-generated by workflow; "Generate New Report" button and `POST /api/reports/generate` endpoint removed |

## Workflow: `interviewDataPipeline`

### Trigger

The interview route handler calls `triggerDataPipeline(session)` as a fire-and-forget async call after:
- `POST /api/interview/:sessionId/pause` вҖ” new endpoint, pauses session and triggers pipeline
- Interview completion вҖ” when all selected broad areas are fully covered (all sub-areas reach `covered` status)

**Not** triggered on every individual answer submission вҖ” that would fire the pipeline too frequently. The user must explicitly pause or complete the interview to trigger report generation.

The function:
1. Gets the session's `selectedBroadAreas`
2. For each broad area, checks coverage (any sub-area with status `covered`)
3. For qualifying areas, checks current report status:
   - No report or `failed` вҶ’ create record as `generating`, start workflow
   - `ready` вҶ’ stash current `content` into `previousContent`, set status to `generating`, start workflow (user can still preview/download previous version)
   - `generating` вҶ’ set `pendingRegeneration: true` on the report record, skip workflow start
4. Starts the workflow with session data and list of qualifying broad areas

**Performance:** Lane B (metrics) completes fast (no LLM calls) so Dashboard, Process Analysis, and SME Engagement pages update almost immediately. Reports follow as LLM work completes in Lanes A, C, D.

### Step Details

#### Step 1: collectSessionData
- Fetch full session from OpenSearch (`consultant_conversations` index)
- Extract: all Q&A grouped by sub-area, coverage map, conversation context (gaps, pain points, opportunities)
- Output: `SessionData` object passed to all downstream lanes

#### Lane A вҖ” Broad Area Reports

**Step 2a: mapSubAreaSummaries**
- For each broad area with coverage, get its sub-areas
- `Promise.all`: for each covered sub-area, call LLM to summarize Q&A into structured digest:
  - Key findings, pain points, maturity level, identified gaps, transformation opportunities
- Uses existing `summarizeCategoryForGap()` pattern from `reportService.ts`

**Step 2b: reduceIntoAreaReports**
- For each broad area, synthesize all sub-area digests into a gap analysis report:
  - Gaps by category (process, technology, capability, data)
  - Current state вҶ’ target state per gap
  - Impact/effort ratings, fit assessment
  - Quick wins, remediation roadmap, risk assessment
  - Chart data: heatmap, Sankey, treemap, gauge charts
- Uses existing `generateGapReport()` pattern from `reportService.ts`

**Step 2c: saveAreaReports**
- Upsert each report to `consultant_reports` (keyed by `sessionId + broadAreaId`)
- Set status: `ready`, calculate file size, set `updatedAt`
- Clear `previousContent` (new content replaces it)
- Broadcast `report-status` SSE event per report

#### Lane B вҖ” Metrics

**Step 3a: computeDashboardMetrics**
- Call existing `recomputeAndStoreMetrics()` from `metricsService.ts`
- Computes: gap severity, critical issues, automation quotient, discovery progress, process flow, process type distribution, process efficiency

**Step 3b: computeSMEEngagement**
- Query all interview sessions across all users
- For each user, compute real engagement metrics:
  - Participation rate: sessions participated / total sessions
  - Response count: actual answers submitted
  - Coverage: broad areas and sub-areas covered
  - Last active timestamp
  - Engagement score: weighted composite of above
- Store in `consultant_sme_engagement` index (new)

**Step 3c: broadcastMetrics**
- Push `dashboard-metrics` SSE event (existing format)
- Push `sme-engagement` SSE event (new)

#### Lane C вҖ” Session-Level Reports

**Step 4a: generateSessionReports**
- Using area reports from Lane A as input:
  - **Executive Summary (readiness)**: Score each broad area (0-100), maturity level, strengths/weaknesses/recommendations. Uses existing `generateReadinessReport()` pattern.
  - **Consolidated**: Merge readiness + gap analysis. Uses existing `generateConsolidatedReport()` pattern.
  - **Strategic**: Automation opportunities, implementation roadmap with tasks/dependencies/owners, ROI projections.

**Step 4b: saveSessionReports**
- Persist each to `consultant_reports` (keyed by `sessionId + type`)
- Broadcast `report-status` SSE events

#### Lane D вҖ” Insights

**Step 5a: computeInsights**
- Using area reports from Lane A:
  - Trend data: compute monthly trends from historical sessions (total, completed, scores over time)
  - AI gap analysis: summarize top gaps across all broad areas with severity and impact
  - AI automation opportunities: identify top automation candidates with estimated savings
- Replaces hardcoded placeholder actions in current Insights page

**Step 5b: saveInsights**
- Store in `consultant_insights` index (new)
- Push `insights-updated` SSE event

#### Final Step: checkPendingRegeneration
- Query all reports for this session where `pendingRegeneration: true`
- If any found: clear the flag, re-trigger the workflow with fresh session data
- If none: workflow complete

## Data Model Changes

### Modified: `consultant_reports` index

New/modified fields:

| Field | Type | Purpose |
|-------|------|---------|
| `broadAreaId` | keyword | Links report to a broad area (e.g., `"o2c"`). Null for session-level reports. |
| `broadAreaName` | text | Display name (e.g., `"Order-to-Cash"`) |
| `pendingRegeneration` | boolean | Queued flag вҖ” true if new data arrived during generation |
| `previousContent` | object | Stashed previous report content while regenerating, so users can still preview/download the prior version |
| `updatedAt` | date | Timestamp of last status change |

New report `type` value: `"broad_area"` alongside existing `readiness`, `gap_analysis`, `consolidated`, `strategic`.

### New: `consultant_sme_engagement` index

| Field | Type | Purpose |
|-------|------|---------|
| `userId` | keyword | User identifier |
| `username` | text | Display name |
| `role` | keyword | User role |
| `department` | keyword | Department |
| `engagementScore` | float | 0-100 composite score |
| `participationRate` | float | Sessions participated / total |
| `responseCount` | integer | Total answers submitted |
| `broadAreaCoverage` | object | Per-area coverage percentages |
| `lastActive` | date | Last activity timestamp |
| `updatedAt` | date | Last computation timestamp |

### New: `consultant_insights` index

| Field | Type | Purpose |
|-------|------|---------|
| `sessionId` | keyword | Associated session (or `"global"` for cross-session) |
| `trendData` | object | Monthly trend arrays (sessions, completions, scores) |
| `gapAnalysis` | object | AI-generated top gaps with severity/impact |
| `automationOpportunities` | object | AI-generated automation candidates with estimated savings |
| `recommendedActions` | nested | Prioritized action items with type, description, impact |
| `computedAt` | date | When this insight was generated |

## Mastra Workflow File

New file: `backend/src/mastra/workflows/interviewDataPipeline.ts`

Registered in `backend/src/mastra/index.ts` alongside existing agents.

The workflow uses Mastra's `createWorkflow` API with step definitions. Each step receives the output of the previous step(s) via the workflow context. Parallel lanes are implemented with `Promise.all` inside step handlers.

## Error Handling

| Failure | Behavior |
|---------|----------|
| LLM fails for one sub-area in map step | Skip that sub-area, proceed with others. Report marked `ready` with a `warnings` array noting which sub-areas failed. |
| LLM fails in reduce step (area report) | That area's report marked `failed`. Other areas unaffected. User can retry from UI. SSE broadcasts failure. |
| Session-level report generation fails | Marked `failed` independently. Area reports remain `ready`. |
| Metrics computation fails | Log error, skip. Dashboard shows stale data until next successful run. |
| Insights computation fails | Log error, skip. Insights page shows stale data. |
| OpenSearch write fails | Retry once. If still fails, log error, report stays `generating`. |
| SSE connection drops | Frontend `EventSource` auto-reconnects. On reconnect, fetches current state via `GET /api/reports` to sync. |

### Stale Report Cleanup

On server startup, scan for reports with `status: "generating"` and `updatedAt` older than 15 minutes. Mark them as `failed`. This handles server restarts mid-workflow.

## Testing Strategy

| Layer | What to test |
|-------|-------------|
| **Workflow steps** | Each step in isolation вҖ” mock OpenSearch reads, mock LLM calls, verify correct input/output shape between steps |
| **Trigger logic** | `triggerDataPipeline` вҖ” deduplication (skip when generating + set pendingRegeneration), coverage filtering (only areas with covered sub-areas), queuing behavior |
| **Parallelism** | Verify Lane A and Lane B run concurrently; verify Lane C and Lane D wait for Lane A |
| **SSE endpoints** | Events pushed on status changes; reconnect delivers current state |
| **Frontend** | Banner appears/disappears; row states transition correctly; SSE events update local state; previous version accessible during refresh |
| **SME Engagement** | Real per-user metrics match session data; SSE updates render |
| **Insights** | AI-generated actions replace placeholders; trend data matches historical sessions |

## Out of Scope

- Right sidebar (Key Risks & Narratives, SME Engagement Heatmap) on Reports page
- Changes to the interview flow itself (question generation, answer submission UX)
- BullMQ/Redis job queue (revisit if concurrent session volume requires it)
